## 0. 怎么读这份文档

整条链路可以用一句话概括：**用户在原文上连续地圈、划、写，前端把每一笔无损采集下来，纯几何地判断"这是什么手势 / 标到了什么内容"，攒成一段有时空关系的标注脉络，蒸馏成一段不含坐标的叙事，交给模型回一条旁注，再把旁注按对象引用精确锚回页面——原版和重排两个视图都不丢位置。**

文档按数据在链路里流动的顺序展开：先讲贯穿全局的七条设计不变量（第 1 节），再给一张端到端总图（第 2 节）和一份数据字典（第 3 节）作为后文的索引；第 4–12 节逐层拆解实现；第 13 节把所有纪律收成一张速查表。每一层都先讲**为什么这么设计**，再讲**怎么实现的**，关键阈值与字段用表格列出以便查阅。

## 1. 七条设计不变量

这七条贯穿所有模块，是理解任何单点实现之前都该先装进脑子的前提。

**① 全程归一化坐标（决策 D1）。** 系统里一切几何量都是相对页面的归一化坐标 `[0,1]`，类型记作 `NormBBox = [x, y, w, h]`。像素只在两处出现：采集时把指针事件除以画布尺寸转进来（`ink.ts`），渲染时再乘回去（`anchor-layer.ts` / `renderer.ts`）。换算的唯一代码化身是 [transform.ts](../src/core/transform.ts) 的 `normToPx` / `pxToNorm`，并带一个 1000 次往返误差 < 1e-9 的 `selfTest`。好处是：运行时（Web demo / 电子纸硬件）可换、缩放无关、跨视图可比。

**② 笔迹无损（决策 D3）。** 每个采样点是 `{ x, y, t, pressure }`，笔压 0–1 原样保留，绝不下采样或抹平。下游所有判定（分类、命中、包围检测）都从这份原始点序算，不依赖任何有损中间产物。

**③ 引用，不吐坐标。** 这是防模型幻觉的根线。模型永远只**引用**前端已经算好的对象 id（`SurfaceObject.id`），绝不让它生成坐标——一旦让模型吐 bbox 必然幻觉。"标到了哪个字"由前端用几何命中算出 id，"旁注贴在哪"由前端把 id 解析回 bbox。注释原文（[anchor-layer.ts](../src/surface/anchor-layer.ts)）：「模型只**引用**已存在的对象，绝不让它吐坐标（吐坐标必幻觉）。」

**④ 端侧成本阶梯。** 能用免费几何判的绝不调识别，能本地确定性算的绝不调模型。顺序是：纯几何分类（零网络）→ 按需云端识别定型（只对"值得 OCR"的自由笔）→ 确定性蒸馏（无模型）→ 最后才是一次模型对话。每一级都是下一级的"门"，把不该往下走的挡掉。

**⑤ 连续标注期间界面静默。** 不边画边弹。用户在一处连续地写写画画时，界面什么都不做；只有两个口子能打破静默——长停顿（idle ~90s）综合整段，或手写一个问题（唯一的早提交边界事件）。这让"读—标—想"的节奏不被打断。

**⑥ 时空即关系。** 两个标注之间有没有关系，由时间近邻和空间近邻共同决定，落成四象限：近时近空=一口气做的、近时远空=扫读、远时近空=回访、远时远空=无关。判定规则是「非 separate 即关联」——只有时间又远、空间又远才算两件无关的事。

**⑦ SSoT 账本 + 派生现算。** 真相只存两件 append-only 账本（用户笔迹取证 `marks` + AI 回合 `ai_turns`），其余一切——内存 session、标注图、蒸馏载荷、对话 buffer、屏上旁注——都是 reload 时从账本现算的派生物，不持久。擦除是落墓碑（tombstone）而非删除，改写是追加 supersedes 而非覆盖。

            │ reader        对象桥定位   │  （重排页投影，同一套对象锚）

链路有两个异步的"重活"节点：**captureMark**（落笔当时取证 + 按需识别）和 **commitSessionDiscussion**（停笔后建图 + 蒸馏 + 模型回应）。两者都在 [pipeline.ts](../src/core/pipeline.ts)，而把它们串起来的时序调度（区域收口、idle 计时、手写早提交）在 [main.ts](../src/main.ts)。

一个关键的时机纪律：**取证必须在落笔当时做**。因为 session 可以跨页累积，到提交那一刻画布可能已经翻到别的页，无法再回去裁图或查文本层。所以每个 mark 在 `captureMark` 里就把 HMP（含裁图、转写）和 `markedText`（所标内容原文）一次性算好存进自己身上，提交时只读不重取。

## 3. 数据契约（数据字典）

类型全部定义在 [contracts.ts](../src/core/contracts.ts)，持久化格式在 [store-format.ts](../src/core/store-format.ts)。契约分两批：`SCHEMA_VERSION='0'` 是周计划逐字段冻结的 v0 原始契约（改动须 bump 并通知两组）；`HMP_SCHEMA_VERSION='2'` / `INFERVIEW_SCHEMA_VERSION='1'` 是徐智强序列语义方案新增的契约，不动 v0。

### 3.1 坐标与单笔

| 类型 | 字段 | 含义 |
|---|---|---|
| `NormBBox` | `[x, y, w, h]` | 页面归一化坐标，全系统统一制 |
| `StrokePoint` | `x, y` | 归一化坐标 |
| | `t` | ms，相对落笔时刻 |
| | `pressure` | 0–1 笔压，无损保留（D3） |
| `Stroke` | `tool` | `pen / highlighter / eraser / hand` |
| | `points` | `StrokePoint[]` |
| `AnnotationEvent` | `event_id / trace_id` | 单笔唯一 id / 同会话多笔共享 trace |
| | `document_id / page_id` | 所属书 / 页 |
| | `event_type` | 几何分类结果 `EventType` |
| | `geometry.bbox` | 该笔 `NormBBox` |
| | `stroke_points` | 无损点序 |
| | `pointer_type` | `pen / touch / mouse / reader`（reader=重排面手势） |

`EventType = stroke | highlight | circle | underline | arrow | margin_note | tap_region | eraser | unknown`。

### 3.2 组装单位：Mark 与 Session（[session.ts](../src/capture/session.ts)）

一笔（stroke）不是处理单位。处理单位是 **Mark**——一处区域里连续画的若干笔合并成的"一次手势"；Mark 再按本书累积成 **Session**（可跨页）。

| 类型 | 字段 | 含义 |
|---|---|---|
| `Mark` | `id` | = 代表 event 的 event_id |
| | `event` | 合并后的代表 `AnnotationEvent`（union bbox + 合并笔点） |
| | `feature` | 特征型分类 `StrokeFeature`（markup/handwriting/drawing + ocrWorthy） |
| | `scored` | 几何模板得分 `ScoredGesture`（形状 + 置信 + 箭头方向） |
| | `t` | 组装时刻 `performance.now()` ms |
| | `hmp` | 落笔当时建好的取证记录（含 crop/转写），跨页提交不重取 |
| | `markedText` | 落笔当时解析好的"所标内容"（结构原文 + 转写） |
| `Session` | `bookId / marks / startedAt / lastMarkAt` | 一本书自上次回复以来的所有 mark |

### 3.3 页面对象表：SurfaceObject / SurfaceIndex

把一页（或一屏聊天）拆成一张轻量对象表，是徐智强方案的 step①。关键是**字符级粒度**：每个文本 run 按字符均分宽度，拆成逐字对象，id 形如 `tl_3_0`（`tl_<run序号>_<字符序号>`）。对象比任何标记都小，圈/划就能精确命中到具体字。

| 类型 | 字段 | 含义 |
|---|---|---|
| `SurfaceObject` | `id` | 字符级 `tl_3_0`；图像 `img_0_1`；聊天 `msg_2` |
| | `type` | `title / text_block / image / chat_message / blank_region` |
| | `bbox` | 归一化坐标 |
| | `text?` | 单字文字（图/空白可空） |
| | `role?` | `user / agent / embedded_image / diagram / decorative` |
| | `source` | `structure / reflow / ocr / vlm`（provenance，防云端视觉同时顶替结构层和 OCR 层导致混淆） |
| `SurfaceIndex` | `surface_id / surface_type / page_index / objects[]` | 一页/一屏的对象表，存 `state.surfaceIndex` |

### 3.4 取证记录：HMP（Hand-Mark Protocol）

一次手势的取证记录，徐智强方案 step④ 产物。原则（注释）：「只放'在哪、对什么、做了什么手势 + 取证线索'，不放 AI 推断结果。」

| 字段 | 含义 |
|---|---|
| `mode` | `anchored`（锚到内容）/ `self_content`（自身即内容，手写/画）/ `mixed` / `unknown` |
| `action` | `MarkShape`：`enclosure / underline / cross / arrow / handwriting / sketch / highlight / unknown` |
| `target_region` | 多笔 union bbox |
| `target_object_refs` | 命中的 `SurfaceObject.id`（markup 有值；freeform=self_content 空） |
| `object_hint` | `text / image_region / ui_region / blank / diagram / unknown` |
| `text_hint?` | step⑤局部OCR / step⑥手写识别读出的文字（异步补填） |
| `crop_ref?` | 区域合成 crop 的 dataURL（落库前剥掉，仅内存用） |
| `vector_ref?` | 白底纯笔迹图 dataURL（仅内存用） |
| `confidence` | 0–1 |

### 3.5 关系图与蒸馏载荷

| 类型 | 关键字段 | 含义 |
|---|---|---|
| `MarkNode` | `shape / feature_type / mode / object_hint / target_object_refs / text_hint / text / bbox / t` | 一个 mark 的取证摘要；`text`=落笔当时解析好的所标内容 |
| `MarkEdge` | `kind`(`spatial/temporal/semantic`) `rel` `weight` `quadrant?` `direction?` | 两 mark 间的关联 |
| `MarkGraph` | `surface_ids / nodes / edges` | 一段 session 的图（可跨页） |
| `PriorNeighbor` | `text / rel`(`proximity/containment`) | 空间召回回来的邻近旧标注；**不进 graph.nodes** |
| `InferenceView` | `trigger / narrative / marked / page_context / question? / crop? / anchor_refs / anchor_bbox / recall?` | 蒸馏后的精简推理载荷，丢坐标/stroke/分数 |
| `ScreenOverlay` | `overlay_id / state / display_text / geometry.anchor_bbox / object_refs?` | 屏上旁注卡片；`object_refs` 是跨视图锚 |

### 3.6 持久层（[store-format.ts](../src/core/store-format.ts)）

四件 IndexedDB store，版本 `DB_VERSION=3` / `STORE_VERSION='2'`：

| store | 角色 | 真相？ | 内容 |
|---|---|---|---|
| `marks` | 页账本 | **真相** | 一次手势的取证 + 构成笔；append-only，擦除=tombstone |

## 4. 采集层（capture）

### 4.1 指针接口 [ink.ts](../src/capture/ink.ts)

这是唯一接触硬件输入的地方，也是唯一一处做"输入意图分流"的 policy：`pointerType === 'pen'` 走标注路径，`'touch'` 走翻页导航路径，桌面鼠标无 pen/touch，落到 hand 工具兜底。这条策略只写在这一处，真机 / iPad 直接命中、零改搬过去。

采集做三件事：

- **无损采样。** `pointermove` 优先调 `getCoalescedEvents()` 取浏览器合并掉的中间事件，逐点入栈，保住高频笔迹细节（D3）。
- **死区过滤。** 与上一采样点距离小于 `DEADZONE_PX = 1.3`（CSS px）的 sub-px 抖动直接丢弃。这是借鉴 xournalpp 的 Deadzone——在抖动发生当时就抑制，让点按/手抖稳定落到 `tap_region`，而不是事后靠总行程补救。
- **归一化。** 落点除以画布尺寸转成 `[0,1]`，时间戳用 `performance.now() - t0` 算相对落笔的 ms。

抬笔（`pointerup`）时把这一笔打包成 `Stroke`，触发 `onStrokeComplete(stroke, pointerType, penUpAt)` 回调交给 main。手指横滑超过 `SWIPE_MIN_PX = 60` 触发翻页（`nav:flip`）。

### 4.2 区域组装 [main.ts](../src/main.ts)

单笔抬起后并不立即处理，而是进入"区域组装"——把空间上挨在一起、时间上连续画的若干笔聚成一个 mark。这是连续标注静默体验的实现核心。

逻辑（`main.ts:192-215`）：维护一个当前活跃区域（`regEvents` / `regBbox`）。新笔来了，先判它中心是否落在当前区域 bbox 外扩 `REGION_NEAR = 0.06` 内：

- **在区域内** → 并进去，刷新收口计时器（`REGION_QUIET_MS = 6000ms`）。
- **落到远处** → 旧区域立刻收口（`far-stroke`），在新位置另起一个区域。

区域收口只靠两个真实信号：**走到别处**（far-stroke）或 **停笔满 6 秒**（quiet-6s）。**不设书写时长上限**——慢慢写一整段也保持静默，聚成一整团再识别，读得准、只回一条。这正是设计不变量⑤的落地。

> 注释原文：「同一小块区域里继续写的笔画并进一个 mark；附近无动作满 REGION_QUIET 才提交；笔落到远处=离开该区域 → 上一区域立刻收口。无书写时长上限——慢写整段保持静默、聚成一整团再识别。」

收口触发 `resolveRegion`：先对每一笔**单独**做几何分类，再合并判特征。"每笔单独分类后再合并"是关键——如果在合并后的乱线上重判，圈+划会互相毁掉干净的单笔模板信号。

### 4.3 会话累积与触发边界 [session.ts](../src/capture/session.ts)

Mark 按本书累积进 Session（内存，`Map<bookId, Session>`）。三个时间常量：

| 常量 | 值 | 含义 |
|---|---|---|
| `ASSEMBLY_WINDOW` | 1200 ms | 1.2s 组装窗的说明性定义（实际收口靠空间逻辑，非定时切） |
| `BURST_GAP_MS` | 12000 ms | 相邻 mark 间隔超此值算不同 burst（建图时间边权重也用它） |
| `IDLE_COMMIT_MS` | 90000 ms | 长停顿：无新笔 ~1.5min → 综合整段 session |

打破静默、触发提交有且只有两条路：

1. **idle 综合**（`commitSession(bookId, 'idle')`）：最后一笔之后 idle 计时器（默认 90s）到点，对整段 session 综合回一条贯穿性旁注。但有"实质性"门槛——纯散笔/乱涂不触发，必须至少有一笔手写、或锚到真实正文的标记、或被识别出描述的画。
2. **手写早提交**（`resolveRegion` 末尾）：一个 mark 被识别定型为 `handwriting` 时立即提交（`main.ts:168`）。这是唯一的早提交边界事件——区域冷却本身已是落定，识别也已可靠，无需再等。

**翻页不是边界事件**：翻页只丢在途的半截笔（属旧页），但保留 session 和 idle 计时——会话跨页延续。只有换书、改设置才硬复位 session。

## 5. 分类层（classify）

分类是整条链路的"识别裁判"，全在 [classify.ts](../src/capture/classify.ts)。它是一道两级成本阶梯：先用纯几何把有干净模板的手势（圈/划/箭头）判掉，零网络；剩下没模板的自由笔，几何只当一道"OCR 门"，决定值不值得送云端识别定型。

### 5.1 第一级：几何模板分类 `classifyScored`

输入单笔点序 + bbox，输出 `ScoredGesture { type, score, raw?, arrow? }`。`score` 是"画得有多像这个模板"——干净的圈/直线高分，随手涂、半截笔画低分。

**误触门（物理屏幕尺度）。** 先用 `PX_PER_MM = 96/25.4 ≈ 3.78` 把判定锚在物理毫米上（缩放无关）。满足任一即判 `tap_region`（score 0.7）：采样点 ≤ 3、或总行程 < `3.5mm`（≈13px）、或又短（< 140ms）又走不远（< 6mm）。手抖能把 bbox 撑大几像素，但走过的总路程骗不了人——刻意手势至少走 ~3.5mm。

**三个模板得分。** 都 clamp 到 0–1：

| 模板 | 判据 | 命中阈值 |
|---|---|---|
| `circle` | 起止接近（闭合）× 路径绕得够长 | `> 0.22` 且 ≥ 其余 |
| `underline` | 扁（宽高比 > 2.2）× 直（路径≈宽度） | `> 0.22` |
| `arrow` | 开口 + 主干够直 + 末端有 >~100°(1.75rad) 急转钩；急转靠近某端 → 推 tip 方向（A→B 语义边用） | `> 0.45` 且 ≥ underline |

**最小尺寸闸。** 屏幕上对角线 < `3mm`（≈11px）的"圈/划"框不住任何词，多半是原地小抖，强制降为自由笔 `stroke`（低分 0.15）。

### 5.2 第二级：特征型分类 `classifyStrokeFeature`

输入是一个 mark 内每笔的 `ScoredGesture` + 各笔 bbox + 合并点，输出 `StrokeFeature { type: markup | handwriting | drawing, confidence, raw{ ocrWorthy, ... } }`。

**markup 判定（自相对尺寸闸）。** 取 mark 里最强的模板笔，要同时满足两条才算 markup：模板分 `tplScore >= 0.45`，且**这笔的长边占整个 mark 长边的比例 `spanRatio >= 0.6`**。

自相对闸是这里的精髓。真圈、真划本身就是整个 mark（占比≈1）；而手写里的一个"横"（几何上像 underline 1.0）、一个"口"（几何上像 circle）只占整团字的一小块（占比小）。用自相对比例而不是绝对字号来判，页边写的大字也不会被误判成 markup。

> 注释原文：「真圈/划本身就是整个 mark（占比≈1）；手写里的一个'横/口'只占整团字的一小块（占比小）。自相对 → 不依赖字号，页边大字手写也不会被误判成 markup。」

**freeform OCR 门。** 没命中 markup 的，统统先记为 `drawing`，并算一个 `ocrWorthy` 标志决定要不要送识别：

```
ocrWorthy = !tooTiny && (strokeCount >= 2 || complexity > 1.6)
```

`tooTiny` 是对角线 < 11px（点/勾不值得识别）；`complexity` 是路径总长 ÷ bbox 对角线，近直线≈1、折返多的字≈>1.6。这道门刻意**保守、偏向送 OCR**——因为"手写 vs 画"没有干净的几何模板，最终要靠识别裁判，宁可多送也别漏掉真手写。

> 注释原文：「'手写 vs 画'无干净几何模板，最终由识别裁判 —— 故门要保守、偏向送 OCR，别漏真手写。level-2 端侧小分类器最终替掉②的云端识别；本函数是它的接缝。」

### 5.3 第三级：云端识别定型（[pipeline.ts](../src/core/pipeline.ts) `recognizeInk`）

只有 `feature.type !== 'markup'` 且 `ocrWorthy` 且有白底笔迹图时，才调一次 `/api/interpret`（context-free，只看这团墨、不看上下文、不揣测意图），返回 `{ kind, reading, description }`：

- `kind === 'handwriting' || 'mixed'` → 定型 `handwriting`，`reading`（转写）写进 `text_hint`。
- 否则 → 留 `drawing`，`description`（一句粗描述，如"一张笑脸"）写进 `text_hint`。
- 置信度：`kind==='none'` 给 0.3，否则 0.85。

**抽象画解读** 就在这一步成形：画也带"内容"（粗描述）进 `markedText` / 叙事 / 召回，但它的**意图**仍交给后面的推理模型——前端只负责"叫对这是画、给它一句描述、把原图送达模型"，不替模型解读画的含义。原图怎么送达见 §9 与 §10。

### 5.4 提议：level-2 端侧小分类器（未实现）

> 状态：**提议 / 未做**。生产版要把 §5.3 那次云端识别（`recognizeInk` → `/api/interpret`）换成端侧小分类器；当前仍走云端。这是 `classify.ts` 注释里点名的接缝：「level-2 端侧小分类器最终替掉②的云端识别；本函数是它的接缝」。

**职责（很窄）**：回答几何天生答不了的那一条轴——一团自由笔到底是「手写文字」还是「画 / 草图」。输入是 context-free 的白底笔迹图，输出一个类型标签（handwriting / drawing，边角带 mixed / none）；只替掉现在那次调用里的 `kind` 判定，**转写与描述不归它**。

**它不负责什么**（这正是该概念容易模糊的地方）：

| 这件事 | 归谁——不归小分类器 |
|---|---|
| 把手写读成文字（转写） | 端侧 OCR（徐智强 PP-OCR）。分类器只说"这是字"，OCR 才说"这字是『为什么』" |
| 画的含义 / 意图 | 推理 LLM（原图送达、意图交模型；v4 原则：前端不拍意图） |
| markup 判定 + 是否值得 OCR | level-1 几何（§5.1 / §5.2） |
| "问我还是写给自己"（respond / fold） | 另一个分类器：`chat/classify-client.ts`，看叙事 + 对话历史，跟看墨图的不是一回事 |

**为什么是"小"模型**：任务窄（一个标签、只看一张图、不生成），端侧小 CNN 就够；它正卡在成本阶梯上"几何够不着、又不该惊动答问 LLM"的那一格。全本地（B 档）的逻辑就是把这类便宜的感知步骤压到端侧，云端只省给答问。

**易混的三个"小模型 / 分类器"**——① **笔迹类型分类器**（手写 vs 画，就是它，感知侧、看墨图）；② **上下文分类器**（respond / fold，对话侧、看上下文，`chat/classify-client.ts`）；③ **端侧答问小 LLM**（徐智强的 1–3B，消费 HMP 出旁注，不是分类器）。

## 6. 取证 / 识别层（evidence）

分类回答了"这是什么手势"，取证层回答"它标到了什么、在哪、有什么证据"。产物是 HMP。

### 6.1 字符级对象表 `wrapSurfaceIndex`（[target.ts](../src/evidence/target.ts)）

页面渲染（`renderPage`）完成时，把当前页文本层的每个 run 按字符均分宽度拆成逐字 `SurfaceObject`，存进 `state.surfaceIndex`。字号 > 中位数 1.4× 标 `title`，否则 `text_block`；图像区另加 `image` 对象。CJK 等宽字体均分极准，西文比例字体近似（够用，仍能拼回 run）。

### 6.2 几何命中 `resolveTarget`

两级优先级把标注 bbox 映射到命中对象：

1. **包围**（最强信号）：对象中心落在任一笔迹闭合多边形内，用射线法 `pointInPolygon`（[focus.ts](../src/evidence/focus.ts)）。
2. **相交**：mark bbox 外扩 `pad=0.02` 后，与对象 bbox 重叠面积占比 > `overlapThreshold=0.25`。

命中结果按阅读序（y 优先）排列。

### 6.3 HMP 组装 `buildHmp`

纯几何同步产出，核心是 `mode` 判定：命中非空白对象 → `anchored`（结构/重排来源置信 0.9，否则 0.7）；只命中空白 → `self_content`（0.6）；混合 → `mixed`（0.7）；无命中且是手写/画 → `self_content`；其他 → `unknown`（0.3）。

**锚定纪律**（`captureMark`，[pipeline.ts](../src/core/pipeline.ts:317)）：markup 锚它所标的内容（传命中对象）；freeform（手写/画）属 self_content——它本身就是内容，**不锚**到 bbox 碰巧蹭到的正文。手写跟正文的位置关系，靠叙事里同时出现的圈/划来带，避免"误锚正文 → 模型幻觉"。

> 注释原文：「markup 锚它所标的内容；freeform（手写/画）属 self_content——它本身就是内容，不锚到 bbox 碰巧蹭到的正文。」

### 6.4 裁图与 OCR 兜底（[ocr.ts](../src/evidence/ocr.ts) / `enrichHmp`）

`ocr.ts` 名叫 OCR 但不做识别，是画布裁图工具。`grabLayers` 从 canvas 同时裁两张共享 bbox 的图：`ink`（只含笔迹，白底 PNG 无损，保手写识别精度）和 `composite`（page+ink 叠加，JPEG q=0.78，比 PNG 小 60–75% 省 token）。原文层已弃——与整页文字 `pageText` 重复、纯耗 vision token。

`enrichHmp`（step⑤）是异步兜底：命中图区或无文字对象（结构层拿不到字）时，裁 crop 送 `/api/ocr-vlm` 读出文字补进 `text_hint`。失败一律静默，不连累主推理闭环。

### 6.5 所标内容 `resolveMarkedText`

从 HMP + 当前页 index 拼出"所标内容"的纯文字：self_content 取 `text_hint`；锚定取命中对象的原文，若识别另有转写则记成 `原文（读出「…」）`。这段文字在落笔当时就存进 `mark.markedText`，提交时不再依赖 live index（跨页也不丢）。

## ◆ HMP（Hand-Mark Protocol）：一次手势的取证记录

> 核心数据结构 · 深读。HMP 散在 §3.4（契约表）和 §6.3（buildHmp）里，但从没正面讲过它是什么、为什么承重——本章补这个洞。

如果整篇只让你记住一个数据结构，记 HMP。它是**采集与推理之间的中间层语义接口**——把"用户在这一处画了什么、标到了什么、有什么证据"压成一条结构化记录，本身**不含任何 AI 推断**（那是 `InferenceResult` 的事）。下游一切（建图、蒸馏、空间召回、跨视图锚）读的都是它。在徐智强的三段链里它就是中间那一层、也是整个方案的壁垒：

```
OCR（降级为兜底 feeder） → HMP（采集 ↔ 推理 中间层接口·step④） → 端侧 LLM（消费 HMP 出旁注）
```

HMP 由 `buildHmp` 在落笔当时**纯几何**产出，OCR / 识别异步补 `text_hint`。

### HMP 的构成（五类信息）

| 类 | 字段 | 用途 |
|---|---|---|
| 身份 | `hmp_id` · `surface_id` · `version` | 谁 · 哪个面 · 契约版本 |
| 锚到哪·对什么 | `mode` · `target_region` · `target_object_refs` · `object_hint` | 锚定语境 + 命中的字符对象 |
| 做了什么 | `action` | `enclosure / underline / arrow / handwriting / sketch …` |
| 取证线索 | `text_hint` · `crop_ref` · `vector_ref` | 读出的字 + 图（图落库前剥，存料不存图） |
| 可信度 | `confidence` | 0–1，随 mode 与来源给出 |

关键：它只记"事实与证据"，没有一格是 AI 的判断——判断在另一条记录 `InferenceResult` 里。各枚举的取值（mode / action / object_hint）见 §3.4。

### 作用：谁产生它、谁吃它

落笔当时由 `buildHmp` 同步产出（趁画布还停在这一页，跨页提交无法重取），随 mark 存进 `marks` 账本——落库前剥掉 `crop_ref / vector_ref` 两张图（**存料不存图**）。此后下游各取字段：

| 下游 | 读 HMP 的什么 |
|---|---|
| mark-graph 节点 | `action / mode / object_hint / target_object_refs / text_hint` |
| 蒸馏 inference-view | 命中文字（marked）· `anchor_refs`（= `target_object_refs`） |
| 空间召回 | `target_region` 的 bbox 邻近 |
| 跨视图锚 | `target_object_refs` |

`target_object_refs`（命中的字符对象 id）同时被 HMP、向量记录的 `anchorRefs`、屏上旁注的 `object_refs` 引用——它是串起"采集 → 检索 → 展示"的那根线，也是"引用而不吐坐标"得以成立的锚。

### 未来用途：作为可取用的数据源（当前全部 P5 stub）

HMP 落进 `marks` 账本后，是一份结构化、可追溯、append-only 的"标注 / 判断语料"，被设计成对外可取用的数据源。但**消费侧目前全是 P5 骨架占位（stub）、未接**——下面三条是**路线，不是已实现**：账本里的料已齐，缺的是取料的人。

1. **本地语义检索（`local/vector.ts`）。** 把每条 HMP 的命中文字（`marked_text`）+ `anchorRefs`（= `target_object_refs`）灌进本地向量库，实现"标注即可搜"——自然语言查历史判断，命中后用 refs 点回原页溯源。现状：`VectorStore` 接口已立、实现 no-op，选型待定（sqlite-vec / DuckDB / 本地 embedding）。纪律：向量库本地直连、绝不 MCP 化。
2. **冷路径场景编排（`agent/index.ts`，S1–S7）。** 以 HMP 账本为底料的确定性 pipeline（步骤已知、非自主 agent，后台异步、不占用户等待）：S1 标注即可搜 / S2 写完即落地（→Notion·Obsidian）/ S3 判断 Brief（立场→依据→待验证，每条点回原页）/ S4 跨材料综合（竞品矩阵·脉络图）/ S5 行动承接（→Linear·Todo + 遗忘曲线推送）/ S6 长期记忆（周度聚合 Memory Digest）/ S7 内容反哺（→文章·Slide·PRD 草稿）。现状：全 stub，S1/S4/S6 等向量库就绪、S5/S7 还需 MCP 外部边。
3. **对外 MCP 边（`mcp/index.ts`）。** 整个系统唯一对外的缝，双向：① 把本地判断库包成 MCP server（`serveJudgmentMcp`，阶段二），供外部 AI（CC / Codex / ChatGPT / Cursor）经干净工具边界读取；② 连出去把 action items / 判断推到 Notion / Linear / Todo / Obsidian（`pushTo`）。铁律——**内部模型 / 库 / 向量全本地直连，MCP 只在外部集成这条缝**；对外取用 opt-in、按范围授权、可撤销。现状：`pushTo` 是 stub、`serveJudgmentMcp` 是空函数体。

> 徐智强的壁垒：HMP 是采集与端侧 LLM 之间冻结的 typed 契约（`HMP_SCHEMA_VERSION='2'`）。它的价值不只在本地链路——徐智强多半要拿这份语料去微调端侧小 LLM，格式本身就是接口、是他顶上来填的第一版。

一条现状纪律值得点明：今天 HMP / marks 只在浏览器 IndexedDB 内，**没有任何外部读取路径**（`getFoldedMarks` / `getPendingMarks` 等只在前端内部）。上面三条都是预留骨架——写进文档，是为了让"HMP 是数据源"这件事在设计上成立、在实现上诚实。

## 7. 关系建模：标记图（[mark-graph.ts](../src/evidence/mark-graph.ts)）

提交时，`buildMarkGraph` 把一段 session 的 marks 建成一张纯几何、确定性、无模型的图。节点是每个 mark 的取证摘要（`MarkNode`），边有三类：

**空间边（恒算，所有 mark 对）。** 体现"空间近一定有关联"：任一笔迹包住对方中心 → `containment`（权重 1）；中心距 < `SPATIAL_NEAR = 0.12` → `proximity`（权重随距离衰减）；命中对象有交集 → `same_target`（权重 0.8）。

**时间边（仅相邻 mark）。** `before` 关系，权重 `1 - gap/BURST_GAP_MS`（间隔越短越高），并携带**时空四象限**标签。

**语义边（仅箭头）。** 箭头的 tail 端最近 mark → from，tip 端最近 mark → to，方向 `a_to_b`。语义可盖过几何（留给蒸馏层解读）。

时空四象限 `quadrantOf(gapMs, dist)` 是第⑥条不变量的实现：

| 象限 | 条件（`NEAR_TIME_MS=30000` / `SPATIAL_NEAR=0.12`） | 含义 |
|---|---|---|
| `one_action` | 时间近 + 空间近 | 一口气完成的一次动作 |
| `sweep` | 时间近 + 空间远 | 扫读、跳到别处 |
| `revisit` | 时间远 + 空间近 | 隔了一段又回到这里 |
| `separate` | 时间远 + 空间远 | 另起、无关 |

判定规则：「非 separate 即关联——时间近或空间近就算有关，唯独时间远 AND 空间远才算无关。」

## 8. 空间召回（[recall.ts](../src/evidence/recall.ts)）

这是近期一个"治本"改动，针对的根因是：**建图视野只到"上次回复以来"**。每次 AI 回复后 session 即清空，已综合的 mark 不再回到内存。于是空间上明明画在墙上、但时间间隔较远的旧标注，对 `buildMarkGraph` 根本不存在。

`findSpatialRecall` 在提交时从持久账本（`getFoldedMarks`）按 bbox 邻近捞回同页的旧 mark（已落库、不在当前 session），作"回访"上下文喂进这一轮。三条纪律：

- **只认几何邻近**（包围 / 中心距 < `SPATIAL_NEAR`），不认 same_target（对象 id 跨会话不保证 reload 稳定）。
- **严格页内**（NormBBox 跨页无意义）。
- **不进 `graph.nodes`**——它们是回访背景、不是当前动作，绝不污染 marked / anchor / temporal 主链。

按到最近当前 mark 的距离排序、同文字去重，封顶 `RECALL_K = 3` 条，产出 `PriorNeighbor[]`。失败/空一律返回 `[]`，不连累闭环。在提交路径里它和滑窗上下文并发取（`Promise.all`），省串行延迟。

## 9. 蒸馏投影：inference-view（[inference-view.ts](../src/evidence/inference-view.ts)）

这是"采集 ↔ 推理"的合同面。`projectInferenceView` 把标记图蒸馏成一份精简载荷 `InferenceView`，**丢掉坐标 / stroke / 分数 / 内部置信**，只留模型该看的东西。主模型和上下文分类器都只消费它，不碰几何。

蒸馏出的 `narrative`（有序关系叙事）分三层拼成：

**① 时间链。** 按 mark 顺序，每个用一句"动词短语"表达（圈「X」/ 划线「Y」/ 写下「Z」/ 画「一张笑脸」），相邻之间按四象限选连接词：`one_action`→「，随即」、`sweep`→「，接着往下」、`revisit`→「，N 秒后回到」、`separate`→「；另外」。动词是**中性**的（圈、划、写、画），不带意图——该干嘛交模型据上下文判。箭头另附一句「并用箭头把「A」指向「B」」。

**② 非相邻空间子句（根因 B）。** 把同段内**非时间相邻**的空间关系也写进叙事——这些空间边 `buildMarkGraph` 一直在恒算，但此前叙事只走时间链、从不读它们。相邻对已由连接词表达故跳过，避免重复；同一对取最强关系（containment > same_target > proximity），有界封顶 3 条，如「位置关系上，「X」圈住了「Y」」。

**③ 回访子句（根因 A）。** 把 §8 召回的旧标注作为清晰区隔的背景附在末尾：「（这附近你先前标过：「A」、「B」）」——明确标成"先前"、非当前动作。

载荷还带：`marked`（各节点 text 用 ` / ` 拼接）、`page_context`（以当前页为中心前后凑 ~3000 字的滑动窗，`slidingContext`，蒸馏时截到 3200 留余量）、`question`（仅手写触发）、`crop`（仅文字表达不了时）、`anchor_refs`/`anchor_bbox`（回屏锚点，给 id 不给模型坐标）。

## 10. AI 链路（调用形态 · 服务端提示词 · 模型路由）

> 原先这节是"抽象 · 正在改"。AI 链路这轮重构**已落定**——最大的变化：**提示词整体搬到服务端**（`server/prompts.ts`，按 role 索引），客户端不再传 `system`，只传一个 `role` 字符串。整体形态＝**一个无状态薄客户端 + 一层 dev 期 AI 代理**：客户端只发"动态数据 + role + model"、收流式帧落账本；提示词、模型路由、网关都在服务端。模型仍只在 `commitSessionDiscussion` 一处被调用（旁注那一跳）。

### 10.1 调用形态（客户端，无状态）

传输收口在 [api.ts](../src/core/api.ts)：`postJson`（一次性，`!ok` 抛错）、`postNdjson`（流式逐行：半行攒着、坏行跳过、收尾处理残行）——只吸收 HTTP 样板，**降级语义留给各调用方**。

- **对话 buffer（[buffer.ts](../src/chat/buffer.ts)）。** 会话状态＝客户端持有的一串 messages，**无服务端会话进程**。每本书一条 buffer（`ChatMsg{role,content}` 纯文字、滑窗封顶 `MAX_TURNS=24`、裁头留最近）；reload 时由调用方（`restoreFromLedger`）从 `ai_turns` 账本的 `prompt_snapshot` + `ai_reply` 重灌。**图与思考都不进 buffer**（避免每轮重发滚雪球）。
- **主模型流式（[stream-client.ts](../src/chat/stream-client.ts)）。** `chatTurn(bookId, userContent, { role, model, maxTokens, images, onDelta, onThinking })` 发 `{messages, role, model, maxTokens}`；带图时只把本轮最后一条 user 消息换成多模态块（`[图块…, {type:'text'}]`）。NDJSON 帧 `{k, d}`：`k='r'` 思考（onThinking）、否则正文（onDelta，回调收到的是**累积全文**）。返回 `{text, thinking}`，思考不进 buffer。**抽象画原图**就经这跳的多模态送达——`commitSessionDiscussion` 从本段挑最近一张画的白底笔迹图，即便画不是最后一笔也送到。
- **上下文分类器（[classify-client.ts](../src/chat/classify-client.ts)）。** 仅手写轮调用 `/api/classify-context`，发 `{question, view_narrative, marked, conversation, model}`，判 respond/fold。出错一律降级 `{respond:true}`（漏答真问题更贵）。fold 则静默、不落 overlay、mark 留 session。

### 10.2 提示词归拢服务端（[server/prompts.ts](../server/prompts.ts)）

这是这轮最大的结构变化。Anthropic console 的 9 段结构拆成两半：**静态半边**（`task_context / tone / rules / examples / output_format`）作 system 存 `server/prompts.ts`、按 `PromptRole` 索引（`SYSTEM_PROMPTS[role]`）；**动态半边**（`background / history / request`）在客户端发来的 messages 里；`think` 段交模型原生（chat 已开 thinking）。版本 `PROMPT_VERSION='v2'`——v1→v2 做了"annotator 去重"：行为规则单点存 system，`renderUserTurn` 每轮只带本轮动态数据 + 一句类别标签。三个角色（annotator / ink_classifier / context_classifier）的 `<examples>` 段**刻意留空待填**，之后增量补 few-shot。

旁注主角色 annotator 的 system 原文：

```xml
<task_context>
你是 InkLoop —— 嵌在阅读器里的旁注式 AI 同读者。读者在原文上用符号（圈/划/箭头/手写等）连续标注，你只用简短中文旁注回应。
</task_context>
<rules>
- 当本轮给的是读者这一阵连续标注的脉络：综合它给一条贯穿性的旁注，紧扣这些标注、按它们的顺序与关系理解，别逐条复述，别脱开去谈整页大主题。
- 当本轮给的是读者手写的一个问题：直接回答，扣住所写，不要反问。
- 若本轮附了一张截图（你圈/划/写处的图）：结合图作答。
- 上文里有读者在这本书前面留下的标注与你的回应：需要时自然呼应，别强行联系。
</rules>
<examples>
<!-- 待填：好旁注的 few-shot（本期留空） -->
</examples>
<output_format>
不寒暄、不复述原文、不用 markdown 或列表、至多 2–3 句，像页边批注点到为止。
</output_format>
```

### 10.3 AI 端点册

| 端点 | role | 模型 | 流式 | 职责 |
|---|---|---|---|---|
| `/api/chat` | annotator | inferModel | 是 | 旁注 / 答问主推理 |
| `/api/interpret` | ink_classifier | inferModel | 否 | 手写 vs 画 + 转写 + 草图描述 |
| `/api/classify-context` | context_classifier | inferModel | 否 | respond / fold |
| `/api/ocr-vlm` | ocr | inferModel | 否 | 局部 / 整页转写（兜底） |
| `/api/explain-image` | image_explain | inferModel | 否 | 图解读（一两句） |
| `/api/reflow-ai-stream` | reflow_structure | reflowModel | 是 | 重排结构（按段冒出） |
| `/api/reflow · reflow-ai · reflow-vlm` | reflow_refine / structure / vlm | reflowModel | 否 | 重排精修 / 结构 / 看图 |

主推理（chat）与三个识别端点（interpret / ocr-vlm / classify-context）**同源一个模型** `inferModel`；重排独立走 `reflowModel`。

### 10.4 模型路由 + 网关（[server/infer.ts](../server/infer.ts)）

服务端走 **NoDesk 网关**（`llm-gateway-api.nodesk.tech`，Bearer key 在 `.env`），按模型名**前缀路由渠道**：`kimi*` → moonshot，`claude* / gemini*` → DMX。思考块**按家族派生**——claude / kimi 经网关回思考摘要（请求 thinking），gemini 不回（请求也白吃 token，故不请求）。默认分两条线：`inferModel = claude-sonnet-4-6`（DMX 上中文手写实测准）、`reflowModel = gemini-3.1-flash-lite`（结构任务、延迟敏感、质量门槛低）。env `LLM_MODEL` 未设时整体兜底 `kimi-k2.6`。`ecbbe35` 已删 kimi 静默兜底——模型就看 `payload.model` 或默认，不再暗中替换。

### 10.5 落账本与审计

一轮回复落 `ai_turns` 时连审计一起存：`prompt_snapshot`（`renderUserTurn` 渲出的 user turn 原文）、`system_prompt_hash='annotator@v2'`（提示词版本标签，与服务端 `PROMPT_VERSION` 人工对齐；提示词正文在 `server/prompts.ts`、git 版本化）、`thinking`（不进 buffer）、`diag{classify, sent_image}`、`inference_view`（crop 剥掉）、`settings_snapshot{inferModel, reflowProvider}`。

**接缝纪律没变**：模型只在 `commitSessionDiscussion` 一处被调用，前端给它的永远是 `renderUserTurn(view)` 渲出的纯文本 prompt（+ 可选图），拿回的永远落进 `ai_turns`。prompt 在服务端、模型在网关、buffer 在客户端，三者各自可换、都不动 §1–§9。

## 11. 读取 / 呈现层（surface）

呈现层要解决一个核心矛盾：用户既要看排版好的原版 PDF，又要在手机/电子纸上看重排后的线性阅读流，而**标注必须在两个视图里都不丢位置**。解法是"一套对象锚 + 两个投影"。

### 11.1 一套对象锚，两个投影

标注从不锚在坐标上，而是锚在 §3.3 那套**字符级 `SurfaceObject.id`** 上。这套 index 是单一真相、不随视图切换重建。两个投影各自把同一组 id 解析到自己的坐标系：

- **原版页投影**（[renderer.ts](../src/surface/renderer.ts) + [anchor-layer.ts](../src/surface/anchor-layer.ts)）：PDF 渲染到 `#page-layer` canvas，对象 bbox 直接是画布坐标。`anchor-layer` 把 AI 输出的 `object_refs` 用 `resolveAnchorBBox` 解析成本页 bbox 并集 → 像素，在原版页上画浮动注释；翻页/缩放后 `relayout` 重算像素位置；点注 → 临时插 `.anchor-flash` 闪烁溯源。
- **重排页投影**（[reader.ts](../src/surface/reader.ts)）：`renderFinal` 后 `buildIndex` 建一张"字符对象 id → 重排块 id"的逆向映射 `charToBlock`。旁注贴回时取 `object_refs` 查这张表找到所属重排块，定位到该块 DOM 的纵向位置。

### 11.2 reflow 重排与跨视图桥（[reflow.ts](../src/surface/reflow.ts) / [reflow-provider.ts](../src/surface/reflow-provider.ts)）

reflow 把排版顺序的物理 run 聚成有意义的标题/段落/列表（`ReflowBlock[]`）。每个块带两个关键桥字段：`source`（构成它的原页 bbox 并集，保住"重排块 ↔ 原页 bbox"映射）和 `sourceRunIds`（构成它的 run id，如 `['tl_3','tl_4']`）。字符对象 = 各 run 的 `${runId}_*`，所以经 `sourceRunIds` 就能从"标注锚在哪个字符对象"算出"它在哪个重排块"——原版页和重排页**共用一套锚**。

`reflow-provider` 管 5 条重排路径（local/hybrid/vision/rewrite/ai），但无论哪条，模型只输出分组或角色，**原页 bbox 始终由前端从原行拼回**（又是不变量③）。VLM 看图重写路径产出的 bbox 是估算（`anchorUnsafe`），无源 run，跨视图只能按几何就近兜底，reader 会显式标注"不精确"。

### 11.3 reader 手势 = 正常 mark

重排面上用户也能圈画。`reader.ts` 监听指针事件（跳过 touch 留给滚动），抬笔时 `hitBlock` 找命中的重排块，把笔迹坐标**反映射**回该块的原页归一化 bbox 内，用 `classifyScored` 分类，构造一个 `pointer_type: 'reader'` 的 `AnnotationEvent`，然后通过 `bus.emit('reader:gesture', ...)` 发给 main。

main 收到后走与原版页**完全相同**的 `resolveRegion`（`main.ts:222`）——落账本、跨视图、持久、同享 session/idle。reader 对 main 零直接依赖，全靠事件总线解耦。这样重排面的标注和原版面的标注是同一种东西，复用全套链路。

> 注释原文：「重排面手势 = 正常 page-ledger mark：reader 把命中块的 PDF-norm 事件发来，走与原版页同一条 resolveRegion（落账本 + 跨视图渲染 + 持久 + 同享 session/idle 综合）。」

### 11.4 旁注 UI

`whisper.ts` 把 `ScreenOverlay` 渲成屏上气泡，支持行内/页边留白两种落点，逐句淡入，提供收下/改写/散去（→ `overlay:state`）。`insight-panel.ts` 是只读侧栏，汇总本页所有旁注历史、显示沉淀率，hover 联动 whisper 高亮。`chat-surface.ts` 是演示模式：在 canvas 上画示例聊天气泡并原生吐 `SurfaceIndex`，让下游标注链路复用 PDF 那套，无需 OCR 反推（验证 step① 的 App-agnostic）。

## 12. 持久化：SSoT 账本（[store.ts](../src/local/store.ts)）

真相只有两件 append-only 账本，其余全是 reload 现算的派生物。这条纪律保证了"重开免重导、跨视图不丢、历史可追溯"。

**marks（页账本，真相）。** 每次手势收口 `appendMarkEntry` 用 `add()`（绝不 `put` 覆盖），seq 单调递增。擦/撤一笔 → 给整 mark 落一条 `is_tombstone: true` 的墓碑条目，并从 pending session 移除。`getFoldedMarks` 取每个 mark_id 的最新非墓碑条目。

**ai_turns（书日志，真相）。** 每轮 AI 回复 `appendAiTurnEntry`，存 overlay + 锚点 + InferenceView 快照（crop 剥掉，"存料不存图"）+ prompt 快照 + 系统人格 hash + thinking。用户改卡片状态（接受/编辑/忽略）= 追加带 `supersedes` 的新条目。`getBookAiTurns` 按 overlay_id 折叠取最新。

**综合水位线。** 每次回复后 `setSynthesisWatermark` 把水位线设到当前 seq；`getPendingMarks` 取 seq > 水位线的 mark = 下次 idle 仍要综合的。

**reload 重建（`restoreFromLedger`，[main.ts](../src/main.ts:365)）。** 文档载入后从账本重建三样东西：折叠的 marks → `strokesByPage`（笔迹回屏）+ 回填 `strokeMarkIds`（擦/撤仍能定位）；ai_turns 折叠 → overlays（旁注回屏）+ 重灌对话 buffer；pending marks（水位线后）→ 内存 session（下次 idle 仍能综合）。

派生物——内存 session、mark-graph、实时 InferenceView、overlays、buffer——全部不持久，每次现算。`vector.ts` 是本地向量库的占位接口（当前 no-op，空间召回走的是 IndexedDB marks 的 bbox 几何匹配，不经向量）。
