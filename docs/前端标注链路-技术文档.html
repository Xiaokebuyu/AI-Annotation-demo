在原文上圈、划、写——一笔从落下到回屏，前端做了什么

用户连续地圈、划、写，前端把每一笔无损采下来，纯几何地判断"这是什么手势、标到了什么"，攒成一段有时空关系的脉络，蒸馏成一段不含坐标的叙事交给模型，再把旁注按对象引用精确锚回页面——原版与重排两个视图都不丢位置。

1七条设计不变量

白话这七条是贯穿全篇的"底层规矩"，后面每一节本质上都是其中某条在某一层的落地。

这七条贯穿所有模块，是看懂任何单点实现之前都该先装进脑子的前提。后面每一节本质上都是它们在某一层的落地。

1
D1 · 坐标

全程归一化坐标

一切几何量都是相对页面的 [0,1]。像素只在采集（除以画布）和渲染（乘回去）两端出现。运行时可换、缩放无关、跨视图可比。

2
D3 · 无损

笔迹无损保留

每个采样点是 {x,y,t,pressure}，笔压原样留。下游所有判定都从原始点序算，不依赖有损中间产物。

3
防幻觉

引用，不吐坐标

模型永远只引用前端算好的对象 id，绝不让它生成 bbox——吐坐标必幻觉。"标到哪个字"由几何命中算 id，"贴在哪"由前端把 id 解析回坐标。

4
成本

端侧成本阶梯

能几何判的不调识别，能本地算的不调模型：纯几何 → 按需云端识别 → 确定性蒸馏 → 最后一次模型对话。每级是下一级的门。

5
体验

连续标注静默

不边画边弹。连续写画时界面什么都不做，只有长停顿综合、或手写一个问题能打破静默。

6
关系

时空即关系

两标注有没有关系，由时间近邻与空间近邻共同决定，落成四象限。规则：非 separate 即关联——只有时间又远空间又远才算无关。

7
真相

账本 + 派生现算

真相只存两件 append-only（只追加、不改不删）账本（marks / ai_turns），其余一切都是 reload（重新打开）时现算的派生物。擦除落墓碑、改写用新条目盖过旧的（supersedes）。

为什么取证要趁早session 可跨页累积，到提交那刻画布可能已翻到别页、无法回去裁图查字。所以每个 mark 在落笔当时就把取证（HMP、转写）和所标内容算好存进自己身上，提交时只读不重取。

4采集层：把笔尖变成区域

白话"采集"就是把你笔尖的动作变成系统能处理的一团团标记。真正的难点不在记录笔画，而在"怎么判断你在某一处画完了"。
为什么是区域不是定时不设书写时长上限。慢慢写一整段也保持静默，聚成一整团再识别——读得准、只回一条。收口只靠两个真实信号：走到别处，或停笔满 6 秒。

ink.ts 是唯一接触硬件输入的地方，也是唯一做意图分流的 policy：pen 走标注、touch 走翻页、鼠标兜底 hand。它做三件事——用 getCoalescedEvents() 无损采样、用 DEADZONE_PX=1.3 在抖动当时就过滤掉比一个像素还小的抖动（sub-px 噪声）、把落点换算成 0–1 的相对坐标（归一化）。

抬笔后单笔并不立即处理，而是进入 区域组装（main.ts）：把空间挨在一起、时间连续画的若干笔聚成一个 mark。下面这个演示走一遍它的收口逻辑——点"下一步"看区域怎么长、怎么收口。

动态演示 · 区域组装收口

Mark 按本书攒进 Session。打破静默、触发提交只有两条路：最后一笔之后 idle 计时器（默认 90s）到点对整段综合回一条；或一个 mark 被识别成 handwriting（手写）时立即触发提交（唯一的早提交边界）。但"提交"≠"必回"——手写轮会先过上下文分类器（respond/fold，/api/classify-context）读这条手写内容、判它是"问我"还是"写给自己"：fold 就静默、不落旁注、这笔留在 session 计入下次综合，只有 respond 才真回一条。翻页不是边界事件——只丢半截笔，session 与 idle 跨页延续。

5分类层：识别裁判与成本阶梯

白话这一步回答一个问题：你画的这团，是圈/划这种"标记"，还是手写文字、还是随手一画？而且要尽量省钱——能用免费的"尺子"量出来的，绝不花钱请 AI 看。
代码原话'手写 vs 画'无干净几何模板，最终由识别裁判——故门要保守、偏向送 OCR，别漏真手写。level-2 端侧小分类器最终替掉云端识别；本函数是它的接缝。

分类是整条链路的"识别裁判"，全在 classify.ts。它是一道两级成本阶梯：先用纯几何把有干净模板的手势（圈/划/箭头）判掉，零网络；剩下没模板的自由笔，几何只当一道"OCR 门"，决定值不值得送云端识别定型。点下面任一节点看那一步在做什么、花多少成本。

点任一节点 ▸ 看它在做什么、花多少成本。

自相对尺寸闸是 markup（圈/划/箭头这类"标记"）判定的精髓：真圈、真划本身就是整个 mark（占比≈1）；手写里的一个"横"（几何像 underline）、一个"口"（几何像 circle）只占整团字一小块（占比小）。用占比而非绝对字号来判，页边写的大字也不会被误判成 markup。识别定型这一步还顺手完成抽象画解读——画也带一句粗描述（"一张笑脸"）进所标内容，但它的意图仍交后面的推理模型，前端只负责"叫对这是画、给它描述、把原图送达"。

代码原话圈注的天职是圈住内容；一个没真圈住任何内容、内部却含其他笔画的圈，更像"脸轮廓 + 五官"。命中则推翻几何 markup、转 freeform 送 interpret 让 VLM 定夺。

markup 不是终判：语义逃生门（markupLooksLikeDrawing · 治"哭脸漏判画"）。上面那条 markup 边有个洞——笑脸/哭脸的脸轮廓本身就是个闭合圆，几何判成 circle、占比≈1 → 定型 markup → 永远跳过识别，"画"被当成"圈"漏掉；更糟的是它 bbox 边角蹭到正文，还会被误锚成圈注（实测锚出 markedText="，制"）。captureMark 为此加了一道语义复判：一个被几何判成 markup 的 circle，若用 pointInPolygon 做真·圈内判定发现它没真圈住任何非空白内容（含图片）、且含多笔（strokeCount≥2），就推翻 markup、当 freeform 送识别让模型定夺"脸 vs 圈"。判据是语义（圈住内容没有）而非几何形状——pointInPolygon 恰好排除 bbox 边角的假命中（哭脸误判的根）。只对 circle 开、要求多笔（孤零空圈不强判），不误伤正经圈注。推翻后 hmpTargets=[] 不再误锚；再由 /api/interpret 按 handwriting / sketch / mixed / none 重新定型：可能是画，也可能是手写或 mixed，并把原图和转写/描述送入后续推理。

代码原话mixed（图+字）两者都留：描述进 markedText/叙事/召回，原图另送推理（按 hasDrawing 放宽）。

识别返回四种 kind，"画+字/符号"这一格别漏。/api/interpret 对一团自由笔回的不是非黑即白的"手写 vs 画"，而是四类：handwriting（清楚的字）、sketch（画/涂鸦/箭头/孤线）、mixed（又有字又有画——写了个词又画了符号、给字配了个箭头/小图）、none（没内容的散点）。mixed 是最容易被"手写 vs 画"二分框漏掉的一格：它仍按 handwriting 定型、但打上 hasDrawing=true，转写和画的描述两者都留（text_hint 记成"字（画：…）"），原图照样送进推理——画的那半不会因为定了手写就被丢掉。易混 这里的 kind=mixed（识别层：字+画同框）和 §6 里 HMP 的 mode=mixed（取证层：既锚到正文、自身又是内容）是两回事，名字撞了、含义不同，别混。

提议 · 未实现level-2 端侧小分类器：把"手写 vs 画"搬上端侧

上面 ② 那次云端识别（recognizeInk → /api/interpret），生产版要换成一个端侧小分类器——这正是 classify.ts 注释里的接缝：「level-2 端侧小分类器最终替掉②的云端识别；本函数是它的接缝」。它的职责很窄：回答几何天生答不了的那一条轴——一团自由笔到底是「手写文字」还是「画 / 草图」。输入是 context-free（不看上下文、只看这团墨）的白底笔迹图，输出一个类型标签（代码侧当前是 handwriting / sketch / mixed / none，再映射成 handwriting / drawing）；只替掉现在那次调用里的 kind 判定，转写与描述不归它。

它不负责的事，恰是这个概念容易模糊的地方：

这件事归谁——不归小分类器
把手写读成文字（转写）端侧 OCR（徐智强 PP-OCR）。分类器只说"这是字"，OCR 才说"这字是『为什么』"
画的含义 / 意图推理 LLM（原图送达、意图交模型；v4 原则：前端不拍意图）
markup 判定 + 是否值得 OCRlevel-1 几何（本节上面那张成本阶梯）
"问我还是写给自己"（respond / fold）另一个分类器：chat/classify-client.ts，看叙事 + 对话历史，跟这个看墨图的不是一回事

为什么是"小"模型：任务窄（一个标签、只看一张图、不生成），端侧小 CNN 就够；它正卡在成本阶梯上"几何够不着、又不该惊动答问 LLM"的那一格。全本地（B 档）的逻辑就是把这类便宜的感知步骤压到端侧，云端只省给答问。现状 仍走云端 /api/interpret，是 v4 记下的"生产 / 端侧承重项、未做"。

实现不止一条路——再留一个几何分支。把"手写 vs 画"全交给小模型不是唯一解。这条轴虽没有圈/划那样的干净模板，几何上仍有可榨的信号：笔画数与尺寸的规整度、是否沿一条基线左右推进、单笔曲率/折返的分布、闭合面积占比、墨迹密度。把这些统计量凑成一个轻判据，能在零网络下先吃掉一批有把握的（一行小字 vs 一个闭合涂鸦），几何拿不准的才落到小模型 / 云端。这与成本阶梯（不变量④：能几何判的绝不调模型）同向，也是全本地（B 档）最该先榨干的一格——两条分支可叠：几何先筛、模型兜底。

易混的三个"小模型 / 分类器"，一并拆开：① 笔迹类型分类器（手写 vs 画，就是它，感知侧、看墨图）　② 上下文分类器（respond / fold，对话侧、看上下文）　③ 端侧答问小 LLM（徐智强的 1–3B，消费 HMP 出旁注，不是分类器）。

6取证 / 识别：标到了什么

白话分类回答"你画的是什么手势"，这一节回答"它到底标到了页面上哪些字、有什么证据"，产物就是一张"取证单"（HMP）。
代码原话markup 锚它所标的内容；freeform（手写/画）属 self_content——它本身就是内容，不锚到 bbox 碰巧蹭到的正文。

分类回答"这是什么手势"，取证层回答"它标到了什么、在哪、有什么证据"，产物是 HMP。页面渲染时把文本层每个 run（PDF 里一段排版文字片段）按字符均分成逐字 SurfaceObject——对象比任何标记都小，圈/划就能精确命中到具体字。命中用两级优先级：对象中心落在笔迹多边形内（叫"包围"，用射线法判点在不在圈内，最强）＞ bbox 外扩相交。命中的是对象 id，不是坐标。

HMP 的 mode 是纯几何判定，决定这次标注是"锚到内容"还是"自身即内容"。识别需要图时，grabLayers 从画布裁两张共享 bbox 的图：ink（白底 PNG，无损保手写精度）和 composite（叠原文 JPEG q=0.78，省 token）。命中图区或无字对象时再走 /api/ocr-vlm 兜底补 text_hint，失败一律静默。

核心数据结构 · 深读

◆HMP：一次手势的取证记录

白话把 HMP 想成一张"取证单"：你在哪一处、对着什么内容、做了个什么手势，外加几条证据线索——只记这些事实，不含 AI 的任何判断。它是整条链路的中转站，下游全靠它。
代码原话只放"在哪、对什么、做了什么手势 + 取证线索"，不放 AI 推断结果（那是 InferenceResult 的职责）。

如果整篇只让你记住一个数据结构，记 HMP（Hand-Mark Protocol）。它是采集与推理之间的中间层语义接口——把"用户在这一处画了什么、标到了什么、有什么证据"压成一条结构化记录，本身不含任何 AI 推断。下游一切（建图、蒸馏、空间召回、跨视图锚）读的都是它。在徐智强的三段链里，它就是中间那一层、也是整个方案的壁垒所在：

构成：五类信息

HMP 的字段按用途分五类。前两类（身份、锚定）由 buildHmp 在落笔当时纯几何产出；取证线索（读出的字、图）由 OCR / 识别异步补填；可信度随之给出。

作用：谁产生它、谁吃它

它在落笔当时由 buildHmp 同步产出（趁画布还停在这一页，跨页提交无法重取），随 mark 一起存进 marks 账本——落库前把 crop_ref / vector_ref 两张图剥掉，存料不存图。此后整条下游都只读它的字段，各取所需：

未来用途：作为可取用的数据源

HMP 落进 marks 账本后，是一份结构化、可追溯、append-only 的"标注 / 判断语料"。它被设计成对外可取用的数据源——但消费侧目前全是 P5 骨架占位（stub）、未接。下面三条是路线，不是已实现：账本里的料已经齐了，缺的是取料的人。

徐智强的壁垒HMP 是采集与端侧 LLM 之间冻结的 typed 契约（HMP_SCHEMA_VERSION='2'）。它的价值不只在本地链路——徐智强多半要拿这份语料去微调端侧小 LLM，格式本身就是接口、是他顶上来填的第一版。

① 本地语义检索（vector.ts）。把每条 HMP 的命中文字（marked_text）+ anchorRefs（= target_object_refs）灌进本地向量库，实现"标注即可搜"——自然语言查历史判断，命中后用 refs 点回原页溯源。现状：VectorStore 接口已立、实现是 no-op，选型待定（sqlite-vec / DuckDB / 本地 embedding）。纪律：向量库本地直连、绝不 MCP 化。

② 冷路径场景编排（agent/index.ts，S1–S7）。以 HMP 账本为底料的确定性 pipeline（步骤已知、非自主 agent，后台异步、不占用户等待）：S1 标注即可搜 / S2 写完即落地（→Notion/Obsidian）/ S3 判断 Brief / S4 跨材料综合 / S5 行动承接（→Linear/Todo）/ S6 长期记忆（Memory Digest）/ S7 内容反哺（→文章/Slide/PRD）。现状：全 stub，S1/S4/S6 等向量库就绪。

③ 对外 MCP 边（mcp/index.ts）。整个系统唯一对外的缝，双向：把本地判断库包成 MCP server，供外部 AI（CC / Codex / ChatGPT / Cursor）经干净工具边界读取；或连出去把 action items 推到 Notion / Linear / Todo。铁律——内部模型 / 库 / 向量全本地直连，MCP 只在外部集成这条缝；对外取用 opt-in、按范围授权、可撤销。现状：pushTo 是 stub、serveJudgmentMcp 是空函数体。

一条现状纪律值得点明：今天 HMP / marks 只在浏览器 IndexedDB 内，没有任何外部读取路径。上面三条都是预留骨架——把它写进文档，是为了让"HMP 是数据源"这件事在设计上成立、在实现上诚实。

7关系建模：时空四象限

白话你连着画的几个标记之间到底有没有关系？这一节用一个朴素规则判断：时间挨得近、或位置挨得近，就算有关系。
恒算的空间边空间边对所有 mark 对都算，体现"空间近一定有关联"：包围（权重 1）＞ 同目标（0.8）＞ 邻近（随距离衰减）。时间边只连相邻、并带四象限标签。

提交时 buildMarkGraph 把一段 session 的 marks 建成纯几何、确定性的图。节点是每个 mark 的取证摘要，边有三类：空间边（恒算）、时间边（相邻 + 四象限）、语义边（箭头 A→B）。

四象限是第六条不变量的实现：两标注间的关系由时间间隔和空间距离共同决定。判定规则——非 separate 即关联：只有时间又远、空间又远才算两件无关的事。拖动下面这个点感受一下：它落在哪一格，蒸馏叙事里就用哪个连接词把前后两笔接起来。

8空间召回：把墙上的旧标注捞回来

白话你回复一次之后系统就"忘了"之前画的——可墙上明明还画着。这一节就是在你下次标注时，把附近那些旧标注重新捞回来当背景。
根因 A每次 AI 回复后 session 即清空……于是"空间临近但时间间隔较远"的旧标注——墙上明明画着——对 buildMarkGraph 根本不存在。

建图视野只到"上次回复以来"。提交时 findSpatialRecall 从持久账本捞回同页旧 mark，作"回访"背景喂进这一轮。当前代码有三条通道：containment（当前笔迹圈住旧 mark 中心）、proximity（中心距 < 0.12）、same_row（同行边注/行旁问题，纵向中心差 < 0.03、横向距离 < 0.5）。它仍严格页内、不认 same_target，并且不进 graph.nodes；每条召回还可带 mark_id 与当时旧 AI 回复 reply，作为背景而非当前动作。

9蒸馏：把图变成一句叙事

白话在把这一段标注交给 AI 之前，先把它"翻译"成一段不带任何坐标的大白话叙事。AI 只看这段话，不碰几何。
采集 ↔ 推理的合同面主模型和上下文分类器都只消费 InferenceView，不碰几何。坐标、stroke、分数、内部置信全部丢掉——这是"引用不吐坐标"在数据上的强制兑现。

projectInferenceView 把标记图蒸馏成一份精简载荷，丢掉坐标 / stroke / 分数 / 置信，只留模型该看的叙事。叙事分三层拼成：时间链（按四象限选连接词）、非相邻空间子句、回访子句；v3 还带指代行、本页旧批注背景与主题召回占位。

10AI 链路：调用形态 · 服务端提示词 · 模型路由

白话和 AI 打交道的这一跳。这轮重构后定下来的样子：客户端只管把"本轮数据 + 用哪个角色 + 用哪个模型"发出去，提示词和模型选择都搬到了服务端。
已定稿AI 链路这轮重构已落定，最大的变化：提示词整体搬到服务端（server/prompts.ts，按 role 索引），客户端不再传 system，只传一个 role 字符串。

整体形态是一个无状态的薄客户端 + 一层 dev 期 AI 代理：客户端只负责把"本轮动态数据 + role + model"发出去、把流式帧收回来落账本；提示词、模型路由、网关都在服务端。模型仍只在 commitSessionDiscussion 一处被调用（旁注那一跳），但它背后这层现在清清楚楚。

10.1 调用形态（客户端，无状态）

传输收口在 core/api.ts：postJson（一次性，!ok 抛错）与 postNdjson（流式逐行，半行攒着、坏行跳过、收尾处理残行）——只吸收 HTTP 样板，降级语义留给各调用方。对话状态就是客户端持有的一串 messages，无服务端会话进程：每本书一条 buffer（ChatMsg{role,content} 纯文字，滑窗封顶 MAX_TURNS=6（最近 3 轮 user+assistant）、裁头留最近），reload 时由调用方从 ai_turns 账本重灌最近 3 轮。chatTurn 发 {messages, role, model, maxTokens}，带图时只把本轮最后一条 user 消息换成多模态块；图与思考都不进 buffer（避免每轮重发滚雪球）。classifyContext 发 respond/fold 五字段，出错一律降级 {respond:true}（漏答真问题更贵）。

10.2 提示词归拢服务端（server/prompts.ts）

这是这轮最大的结构变化。Anthropic console 的 9 段结构被拆成两半：静态半边（task_context / tone / rules / examples / output_format）作为 system 存在 server/prompts.ts、按 PromptRole 索引；动态半边（background / history / request）在客户端发来的 messages 里；think 段交模型原生（chat 端点已开 thinking）。版本 PROMPT_VERSION='v3'——v2→v3 给 annotator 增加静态 <background>，说明本页已有批注/旧回应只是背景，模型只回应当前聚焦；renderUserTurn 也同步前置“本页已有批注”动态背景。三个角色（annotator / ink_classifier / context_classifier）的 <examples> 段刻意留空待填，之后增量补 few-shot。先用一张图看这一轮喂给模型的整体上下文结构（system 静态 ＋ messages 动态），再看旁注主角色 annotator 的 system 原文：

11读取 / 呈现：一套对象锚，两个投影

白话你既想看排版好的原版 PDF，又想看重排后好读的版式，而你画的标注在两边都不能跑位。这一节讲它怎么做到。
代码原话标注锚在字符对象上 → 经此把"在哪个块"算出来，原版页 / 重排页共用一套锚。

呈现层要解决一个矛盾：用户既看排版好的原版 PDF，又看重排后的线性阅读流，而标注必须在两个视图都不丢位置。解法是标注从不锚在坐标上，而锚在字符级 SurfaceObject.id 上；这套 index 是单一真相，两个投影各自把同一组 id 解析到自己的坐标系。重排块靠 sourceRunIds 这道桥反查"标注落在哪个块"。

重排面上用户也能圈画：reader.ts 把笔迹坐标反映射回命中块的原页 bbox，构造 pointer_type:'reader' 的事件，经 bus 发回 main 走完全相同的 resolveRegion——落账本、跨视图、持久、同享 session/idle。重排面的标注和原版面的是同一种东西，零特殊路径。

12持久化：真相与派生

白话系统只把两样东西当"真账"：你的笔迹、和 AI 的回应。其余一切（当前会话、关系图、旁注…）都是每次打开时现算出来的，不存。
代码原话派生不存（reload 从 marks/docs 现算）：session、mark-graph、实时 InferenceView。页保持纯净：marks 只装用户笔迹 + 取证，AI 文本全在 ai_turns。

真相只有两件 append-only 账本，其余全是 reload 现算的派生物。这条纪律保证"重开免重导、跨视图不丢、历史可追溯"。擦除是落墓碑、改写是追加 supersedes；每次回复后推一道综合水位线，水位线之后的 mark 才是下次 idle 仍要综合的。
